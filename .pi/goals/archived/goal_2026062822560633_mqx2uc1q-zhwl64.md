# GNOME CI Pipeline: End-to-End (Albacore & Yellowfin → All Variants)

## Goal

Get all GNOME image builds passing end-to-end in CI, integrate ISO boot verification, and adopt production-quality CI improvements from upstream Bluefin. Start with albacore and yellowfin, then extend to all variants.

## Success criteria

- `build-yellowfin.yml` passes for all GNOME flavors: gnome, gnome50, gnome-hwe, gnome-nvidia, gnome50-hwe, gnome50-nvidia, gnome-nvidia-hwe
- `build-albacore.yml` passes for the same GNOME flavors
- GNOME ISOs build successfully and pass screenshot-based boot verification (QEMU screendump)
- Serial-based e2e (TUNAOS_LIVE_READY marker) runs as a non-blocking check — warns on failure, does not fail the build
- Same GNOME pipeline succeeds for skipjack and bonito
- Bluefin-derived improvements (new actions, testing patterns, prod promotion flows) are adopted

## Boundaries

**In scope:**
- Fix all failing GNOME image builds for albacore and yellowfin
- Integrate ISO boot verification into TunaOS CI workflows (screenshot primary, serial best-effort)
- Port relevant CI/testing improvements from projectbluefin/bluefin and bluefin-lts
- Extend working GNOME pipeline to skipjack and bonito

**Out of scope:**
- KDE, Cosmic, Niri, or other non-GNOME desktop variants
- bootc-installer GUI porting (separate goal)
- Non-GNOME ISO builds
- New GNOME features or package additions
- Kernel serial driver fix (CONFIG_SERIAL_8250=m in almalinux-bootc — separate follow-up)
- Grouper (Ubuntu 26.04) — separate build infrastructure (Containerfile.ubuntu), deferred to follow-up goal

## Constraints

- Must follow existing CI architecture (build-variant.yml → reusable-build-image.yml pipeline)
- ISO verification must not block existing non-GNOME builds
- Bluefin improvements are adopt-what-fits — no forced alignment
- Serial e2e is non-blocking: emits ::warning on failure, continues pipeline

## Ordered steps

1. **Audit** the current GNOME CI failures for albacore and yellowfin — identify every failing GNOME flavor and root cause
2. **Fix base images** first (albacore:base, yellowfin:base) if they're the blocking dependency
3. **Fix stage-2 GNOME builds** (gnome, gnome50) for albacore and yellowfin
4. **Fix stage-3/4 GNOME builds** (-hwe, -nvidia, -nvidia-hwe variants) for albacore and yellowfin
5. **Integrate ISO boot verification** — build ISOs for all GNOME flavors (PkgS2 + PkgS3), verify with QEMU screenshot as primary gate, run serial e2e as non-blocking check. The serial readiness marker (TUNAOS_LIVE_READY) is blocked by CONFIG_SERIAL_8250=m in the upstream bootc base image — the tunaos-live-ready.service fix (StandardOutput=tty) is correct but the kernel can't output to ttyS0 without a built-in serial driver. Tracked for follow-up: rebuild base image or add 8250 to initramfs.
6. **Adopt Bluefin CI improvements** — new actions, testing patterns, prod promotion flows from projectbluefin/bluefin and bluefin-lts
7. **Extend pipeline** — ensure the same GNOME CI passes for skipjack and bonito (grouper deferred)

## If blocked / unclear / failing

Stop and ask. Each failure is a discrete fix; no guessing at root causes. If a blocker requires work outside this goal (e.g., upstream kernel config), document it and move forward with what's achievable.

## Verification contract

Each step: the CI run for that variant+flavor returns green on a workflow_dispatch from main. For step 5: ISO builds pass, screenshot artifacts show a booted desktop. Serial e2e failure = ::warning only, does not block the step.

## Sisyphus reminder

Work patiently and sequentially. No rushing, no unrequested preflight steps, no improvising around blockers.
